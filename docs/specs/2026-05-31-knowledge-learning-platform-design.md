# AI 学习教练 — 方案讨论日志

**日期**: 2026-05-31
**状态**: 方案已确认，MVP 拆分为 MVP-0/1/2，工程规格细化中
**版本**: v2.1

---

## 1. 产品定位

**一句话**: 一个飞书里的 AI 学习教练，把公开视频、碎片内容和个人目标转成每天 15-20 分钟可完成的学习单元。

**核心产品价值**: 用户真正需要的不是保存一堆摘要，而是每天被温和地推着学一点，并且能看到自己在进步。

**当前阶段**: MVP（2 人，飞书内）
**目标阶段**: C 端发布（独立 App/小程序，飞书仅作渠道之一）

**学习领域（MVP 聚焦）**: AI · Python · SQL · 英语

---

## 2. 核心闭环

```
用户画像 → 制定学习计划 → 推送每日内容 → 多轮学习 → 成长追踪
                              ↑
                    视频平台内容(萃取源)
```

## 5. 场景 & 使用流

### 5.1 用户首次使用
1. 用户在飞书打开机器人
2. 选择学习模板（如"零基础入门 Python"）
3. 机器人追问 3-5 个关键问题（当前水平、学习目标、时间预算、学习约束）
4. 系统从课程模板库匹配路线 → LLM 根据画像个性化改写
5. 生成计划：4 周大纲（对外展示）+ 前 3 天具体内容（内部锁定）
6. 用户确认/调整后，计划生效

### 5.2 每日学习
1. 机器人按用户设定时段推送每日学习单元
2. 每日单元根据 `learning_unit_type` 分发不同结构：

| unit_type | 适用领域 | 结构 |
|-----------|----------|------|
| concept_unit | AI/Python/SQL | 概念卡片 → 代码实例 → 微练习 → 总结 |
| coding_unit | Python/SQL | 题目描述 → 代码框架 → 参考答案 |
| reading_unit | 英语 | 短文 → 词汇表 → 理解题 |
| listening_unit | 英语 | 音频片段 → 听写/填空 → 原文对照 |
| speaking_unit | 英语 | 句型模板 → 场景对话 → 开口练习 |
| review_unit | 全部 | 本周知识点回顾 → 薄弱点重测 |

3. 用户完成打卡，反馈信号反馈（太难/太简单/没时间/感兴趣/已掌握）
4. 系统根据反馈调整未来 3 天内容，不改大目标

### 5.3 内容补充（贴链接）*MVP-1*
1. 用户随时贴视频链接到机器人
2. 系统走 ContentIngestion 接口——内容走完整状态机（见第 9 节）
3. LLM 总结内容 → 过 Quality Gate → 评分入库或丢弃
4. 如果入库，判断是否适配当前学习计划 → 纳入推送队列（MVP-2）

### 5.4 内容分发（抖音/小红书）*P1*
1. 系统根据每日学习总结自动生成帖子文案+配图
2. 常规内容：全自动发布
3. 热点/争议/深度内容：生成草稿推送到飞书，用户确认后发布
4. 评论区自动回复（P2）

### 5.5 飞书交互优先级

MVP-0 不依赖飞书云文档，全部通过飞书消息卡片完成：

```
飞书消息卡片
├── 今日学习卡片（concept_card + example + exercise）
├── 操作按钮：完成 / 太难 / 太简单 / 没时间
├── 查看完整笔记按钮 → 展开详细内容
└── 学习状态卡片（/status 命令触发）
```

飞书云文档作为可读知识库，在 MVP-0 稳定后才接入。

## 8. 用户画像

### 8.1 画像字段

| 维度 | 内容 | 说明 |
|------|------|------|
| 基础信息 | 姓名、学习目标 | 转行 / 提升 / 兴趣 / 面试 |
| 知识基线 | 每个领域当前水平 | 零基础 / 入门 / 中级 / 进阶 |
| 时间预算 | 每日可用学习时段+时长 | 如通勤 20min + 午休 15min |
| 偏好画像 | 内容偏好 | 文字/视频、深度/轻量、严肃/轻松语气 |
| 学习约束 | 使用场景限制 | 能否写代码、是否方便看视频、是否需要纯音频/通勤模式 |
| 反馈信号 | 行为信号 | 太难 / 太简单 / 没时间 / 感兴趣 / 不感兴趣 / 已掌握 |
| 成长记录 | 已学内容、打卡天数、薄弱知识点 | 动态更新 |

### 8.2 反馈信号的作用

反馈信号是学习计划能否动态调整的关键。没有它，4 周计划生成后就是"AI 版课程表"——生成一次就死了。系统需要根据反馈信号调整个性化推送，但不频繁修改用户的大目标，避免失去确定感。

## 10. 学习计划引擎

### 10.1 课程模板库

不能让 LLM 从零发明课程。系统预置课程模板库，LLM 只负责根据用户画像做裁剪和个性化。

```
课程模板库（预置）
    │
    ├── Python 零基础路线
    │   变量/类型 → 条件判断 → 循环 → 函数 → 文件处理 → API请求 → 小项目
    │
    ├── SQL 零基础路线
    │   SELECT → WHERE → JOIN → GROUP BY → 子查询 → 窗口函数 → 实战
    │
    ├── AI 入门路线
    │   AI概览 → 机器学习基础 → 神经网络 → LLM原理 → Prompt工程 → Agent
    │
    ├── 英语（技能习得型）
    │   日常词汇 → 场景对话 → 短篇阅读 → 听力训练 → 输出表达
    │
    └── 进阶路线（入门完成后解锁）
```

LLM 负责：
- 根据用户时间预算拆分每日单元粒度
- 根据用户水平跳过已掌握或补充前置知识
- 根据学习约束调整内容形式（通勤模式 → 音频为主，能写代码 → 加入 coding_unit）

### 10.2 学习单元类型定义

| unit_type | 适用领域 | 结构 |
|-----------|----------|------|
| concept_unit | AI/Python/SQL | 概念卡片 → 代码实例 → 微练习 → 总结 |
| coding_unit | Python/SQL | 题目 → 代码框架 → 参考答案 → 解析 |
| reading_unit | 英语 | 短文 → 词汇表 → 理解题 |
| listening_unit | 英语 | 音频片段 → 听写/填空 → 原文对照 |
| speaking_unit | 英语 | 句型模板 → 场景对话 → 开口练习 |
| review_unit | 全部 | 本周知识点回顾 → 薄弱点重测 |

### 10.3 滚动计划机制

- **对外展示**：4 周完整大纲，用户能看到整条路，有确定感
- **内部锁定**：只锁定未来 3 天的具体学习单元
- **触发重排**：用户跳过、加速、反馈"太难"后，只重排未来 3 天内容
- **不变大目标**：不因单次反馈修改整体计划方向

## 11. 视频内容采集 — ContentIngestion 接口

### 11.1 采集策略（按字幕可得性设计，非按 API 一定可取）

YouTube 官方字幕 API 对权限有要求，B站字幕也不是所有视频都有稳定接口。工程上按"字幕可得性不稳定"设计。

**采集降级链**：

```
收到链接
  → 尝试平台原生字幕（CC字幕 / API）
    → 成功？source_type="platform_subtitle", confidence=0.9
    → 失败？尝试第三方解析工具（yt-dlp / bilibili-api）
      → 成功？source_type="third_party", confidence=0.7
      → 失败？提取音频 → ASR 语音转写
        → source_type="asr_transcript", confidence=0.5
```

### 11.2 平台策略

| 阶段 | 平台 | source_type 优先级 |
|------|------|--------------------|
| MVP-1 | B站 + YouTube | platform_subtitle → third_party → asr_transcript |
| P1 | 小红书/抖音 | 用户手动粘贴文案（source_type="manual"），或 ASR 兜底 |

### 11.3 统一接口格式

所有平台统一成一个 `ContentIngestion` 接口：

| 字段 | 类型 | 说明 |
|------|------|------|
| title | string | 视频标题 |
| author | string | 创作者名称 |
| url | string | 原始 URL |
| platform | string | youtube / bilibili / xiaohongshu / douyin |
| source_type | string | platform_subtitle / third_party / asr_transcript / manual |
| content_text | string | 最终使用的文本内容 |
| subtitle_raw | string | 原生字幕文本（如有） |
| transcript_text | string | ASR 转写文本（如有） |
| confidence | float | 0.0-1.0，内容准确度评估 |
| metadata | JSON | {duration, language, view_count, ...} |

> 产品上明确告知用户：字幕可用时优先字幕，不可用时走 AI 转写，转写内容可能有误。每条内容展示 source_type 和 confidence。

## 12. LLM 质量评估 — Quality Gate

### 12.1 定位

这是产品差异化核心。很多工具会总结，但很少有工具会判断"这东西值不值得学"。

不应该说"AI 判断真假"，应该是"AI 做可信度分级和待核验主张提取"，让用户做最终判断。

### 12.2 评分规则

#### 可信度高 (credibility: high)
- 有明确来源、数据、官方文档或论文引用
- 作者身份可信（认证账号、领域专家、机构号）
- 论证链完整：观点 → 论据 → 验证
- 与常识和权威资料无明显冲突

#### 可信度中 (credibility: mid)
- 内容有一定价值，但来源不足
- 偏经验总结、个人观点，缺少外部验证
- 部分结论需要核验

#### 可信度低 (credibility: low)
- 明显的标题党、营销话术、卖课导向
- 大量绝对化表达（"必学""唯一""100%"）
- 缺少论据，关键事实无法验证
- 技术内容无代码示例、无验证过程

### 12.3 自动决策规则

| 可信度 | 适配度 | 决策 |
|--------|--------|------|
| high | matched_domain ≠ "none" | `add_to_plan` — 自动加入学习计划 |
| mid | matched_domain ≠ "none" | `human_review` — 推送飞书让用户确认 |
| mid | matched_domain = "none" | `archive` — 只存档，不入计划 |
| low | 任意 | `discard` — 丢弃 + 告知用户原因 |
| high/mid | 任意 + risk_tags ≠ [] | `human_review` — 有风险标签一律人工确认 |

### 12.4 Quality Gate 输出卡

| 字段 | 取值 | 说明 |
|------|------|------|
| should_ingest | yes / no / human_review | 入库决策 |
| credibility | high / mid / low | 可信度分级 |
| matched_domain | AI / Python / SQL / 英语 / none | 匹配学习领域 |
| matched_stage | 入门 / 进阶 / 复习 / 拓展 / none | 匹配学习阶段 |
| risk_tags | [] | 可选：无来源 / 过时 / 营销化 / 争议性 / 术语混乱 |
| claims_to_verify | [] | 3-5 条待核验主张 |
| recommended_action | add_to_plan / archive / discard | 系统建议 |

## 13. 内容状态机

```
received ──→ extracting ──→ transcribing ──→ summarizing ──→ reviewing
   │             │               │                │               │
   │        抓取失败        转写失败          总结失败         评分异常
   │             │               │                │               │
   └─────────→ error ───────→ error ──────────→ error ──────────→ error
                                                            │
                                              ┌─────────────┤
                                              ▼             ▼             ▼
                                          queued       archived      rejected
                                       (纳入推送)    (只存档)     (丢弃)
```

| 状态 | 说明 |
|------|------|
| received | 用户贴入链接，已记录 URL |
| extracting | 正在抓取平台字幕 |
| transcribing | 字幕不可用，正在 ASR 转写 |
| summarizing | LLM 正在生成总结 |
| reviewing | LLM 正在跑 Quality Gate |
| queued | 审核通过，纳入推送队列 |
| archived | 审核通过但暂时不适配当前计划，存档备用 |
| rejected | 审核不通过，已丢弃 |
| error | 处理失败，记录错误信息 |

### 失败兜底

| 场景 | 兜底策略 |
|------|----------|
| 视频无法解析（链接失效/私有视频） | 返回 "无法访问该视频，请检查链接" |
| 字幕提取失败 + ASR 转写失败 | 返回 "无法获取内容，请手动粘贴文案" |
| LLM 总结失败（API 超时/返回格式错误） | 重试 1 次 → 仍失败则存档原始文本，标记 error |
| Quality Gate 评分异常（JSON 解析失败） | 默认为 `human_review`，不自动决策 |
| 飞书消息推送失败 | 记录日志，下次定时任务重试 |
| 飞书云文档创建失败 | 降级为消息卡片，不阻塞主流程 |

## 15. 已确认决策汇总

| 决策 | 结论 |
|------|------|
| 产品叙事 | "AI 学习教练"，不是"知识总结助手" |
| 目标用户 | MVP: 自己和同事 2人，飞书内。远期: C 端 |
| 学习领域 | AI、Python、SQL、英语（MVP 聚焦） |
| 学习粒度 | 细粒度，每日 15-20min 碎片单元（Duolingo 式打卡） |
| 学习单元类型 | concept_unit / coding_unit / reading_unit / listening_unit / speaking_unit / review_unit |
| 计划机制 | 滚动计划：展示 4 周大纲 + 锁定 3 天 + 反馈驱动重排 |
| 计划创建方式 | 课程模板库 → LLM 个性化改写（非从零生成） |
| 内容来源 | 混合：用户贴链接 + 系统按计划缺口智能补充 |
| 视频采集 | 降级链：平台字幕 → 第三方工具 → ASR → 手动粘贴 |
| 质量评估 | MVP-1 必须带 Quality Gate + 自动决策规则 |
| 内容分发 | P1，不拖慢主产品验证 |
| 架构 | 方案 B：飞书前端 + 独立 FastAPI 后端 |
| 数据存储 | PostgreSQL 存结构化状态，飞书云文档做可读知识库（MVP-0 后) |
| 交互形式 | MVP-0 优先消息卡片，云文档后接入 |
| 成功标准 | 7天连续使用 · 60%完成率 · 10个链接 · 70% QC认可率 |
